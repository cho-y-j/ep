# SKEP v2 — 전문가 UX·자동화 전략 (4주체 기능테스트 기반)

2026-07-11. 4주체(BP·장비공급사·인력공급사·협력사) 실데이터 기능테스트 + 누적 감사(역할/성능·UX·정산·E/D/A·safe-8) 근거.

## 0. 한눈에 (핵심 결론)

- **4주체 전 기능 실데이터 검증 완료.** 신규 협력사(V77)까지 관계생성·자식로그인·본인관리·정산열람·스코핑·심사루프 **실동작 확인**.
- **이번 세션에 이미 자동화/개선한 것**: 후보배지(E)·정산 근무일수 자동파생(D)·선정→배차 초안(A)·서명대기 집계·OT 프리필·견적 단계칩·투입요청 일괄수락·만료 알림 스케줄러·거짓카피 3곳·후보 자동조회·대시보드 처리큐·다이얼로그 통일.
- **"모든 기능 자동화"의 정직한 답**: 반복·이음새·수기입력은 **대거 자동화 가능/완료**. 그러나 **전자서명·서류 컴플라이언스 최종판정·현장 배치 적합성·반려 사유**는 사람이 판단해야 하며, 자동화는 그 **"주변"**(리마인더·프리필·큐·자동초안)만 해야 최고다. 전부 무인화가 목표가 아니라 **"사람은 결정만, 나머지는 자동"**이 목표.
- **지금 최우선(P0)**: 협력사 기능 갭 2건 — 제품 결정만 주시면 바로 구현.

## 2. 협력사(V77) 집중 분석

### 실동작 확인 (PASS)
| 항목 | 상태 | 증거 |
|------|------|------|
| 관계생성(부모 대행) | ✅ | `POST /api/companies/children`(장비공급사)→자식회사+마스터로그인 동시 생성 201. 인력공급사는 403(EQUIPMENT만 부모 가능) |
| 협력사 로그인 | ✅ | 자식 계정 로그인 200(role=EQUIPMENT_SUPPLIER, 자식 company) |
| 본인 장비/인력 관리 | ✅ | 자식이 본인 장비 생성/수정 201/200 |
| 정산 열람 | ✅ | 자식=본인 소유그룹만(dispatched_by_parent:true), 부모=본인+협력사 두 그룹 |
| 데이터 스코핑(selfAndChildren) | ✅ | 부모는 자식 자원 읽음, 크로스회사 403 |
| 심사/빠꾸 루프(미니-BP) | ✅ | 부모→자식자원 보완요청 201+알림, 비자식 403 |

### 온보딩 3경로 실제 상태
- **(a) 협력사 자가 로그인 업로드 — 부분**: 로그인 후 관리는 완전 동작하나, **자격 획득이 자가서비스 아님**(self-signup→enabled=false→시스템 ADMIN 승인 필요, 부모 승인 경로 없음).
- **(b) 장비업체 대행 등록 — 완전(단 갭)**: 자식회사+로그인+장비 대행생성 전부 동작. **갭**: 생성만 되고 **수정/삭제 403**.
- **(c) 링크 발송→협력사 업로드 — 완전**: `POST /api/document-collections`→토큰, 무로그인 `GET /api/collect/{token}` 200(정렬 필수서류), 업로드/제출 동작.

### 기능 갭 2건 (제품 결정 필요)
1. **[중] 대행 생성 가능·수정 불가 비대칭** — 부모가 자식 장비/인력을 **생성은 selfAndChildren 허용, 수정은 본인회사 고정**(`EquipmentService.ensureCanModify`, PersonService 동일). → 자식 로그인 없이 대행등록한 자원은 **부모도 자식도 못 고치고 시스템 ADMIN만 가능**. **결정**: 대행 모델이 "등록만 도와주고 이후 자식이 관리"인가, "부모가 계속 대신 관리"인가? 후자면 `ensureCanModify`를 selfAndChildren로 확장(수술적, QA 필요).
2. **[중] 협력사 자가 로그인 승인 게이트** — self-signup 계정 enabled=false, 부모 승인 경로 없음(usersByCompany는 ADMIN 전용). **결정**: 부모(장비공급사 관리자)가 자기 협력사 계정을 승인/활성화하는 경로를 열 것인가? 열면 경로(a) 완결. (사업자번호 도용 방지 정책과 균형 필요.)

### 기타(저위험)
- **하위공급사 데모 데이터 미시드**: `DemoDataSeeder`에 협력사 시드 없음 → **DB 리셋 시 V77 시연 데이터 소멸**. 시연/재현 위해 시드 추가 권장(안전·additive).
- BP가 자식 자원을 **부모(장비업체) 명의로** 봄 — 설계 기본값, 실소유 노출 여부는 미정(§결정필요).

## 3. "너무 복잡"의 실체 + 이미 해소한 것

**진단(UX 연구)**: 복잡함은 개별 기능이 아니라 **화면 사이 이음새** — 장비 1대 투입에 화면 9개·역할전환 8회·단가입력 3회. + 거짓 안내문 3곳 + 후보 배지 미구현 + 정산 이중입력.

**이번 세션에 해소한 매핑**:
| 불편 | 해소 |
|------|------|
| 거짓 안내(선정=작업계획서 자동/수락=배치) | P0 카피 3곳 수정 |
| 후보 조회 수동 버튼 | 자동조회 |
| 잘못된 자원 선택→작업계획서 단계서 반려 | E 후보 배지(이전투입/만료임박) |
| 배차 9필드 수기 | A 선정→배차 초안(1클릭 확인후발송) |
| 정산 근무일수 이중입력 | D 작업확인서 자동파생(수동 우선) |
| 단가 3회 재입력 | OT 프리필(투입요청) |
| BP가 할 일 흩어짐 | 대시보드 처리대기 위젯(투입요청·서류심사·서명대기) |
| 견적 진행상황 불투명 | 목록 단계칩(선정→배차→서류) |
| 투입요청 다건 수기 수락 | 일괄수락 |

## 4. 자동화 경계 — 전문가 의견 (핵심)

| 구간 | 대상 | 원칙 |
|------|------|------|
| ✅ **자동화 완료/가능(safe)** | 후보배지·근무일수 파생·배차초안·서명대기/처리 큐·OT/단가 프리필·단계칩·일괄수락·만료 리마인더·작업계획서 복제 | additive·읽기/프리필/집계. 검증된 핵심흐름 무변경 → 계속 확장 |
| 🟡 **제품결정 후 자동화** | 협력사 대행수정·자가로그인 승인·C(수락=배치)·A-2(TARGETED 초안)·B(단가원장 단일화)·서류허브 통합·BP 네비 재그룹 | 도메인/의미/데이터모델이 갈림 → **결정 먼저**, 그 다음 자동화 |
| 🔴 **자동화 금지(사람 필수)** | 전자서명 5인·서류 컴플라이언스 최종판정·현장 배치 적합성·반려 사유 판단 | 자동화는 **주변만**(리마인더·프리필·후보추천·큐). 결정 자체는 사람 |

**요지**: "모든 기능 자동화" = **사람은 판단·서명만, 준비·이동·집계·알림은 전부 자동**. 판단까지 자동화하면 컴플라이언스·안전 리스크.

## 6. 주체별 "가장 편하게" 한 줄
- **BP**: 대시보드 처리큐 + 견적 단계칩으로 "지금 뭐 할지"를 한 화면에. 다음은 네비 4그룹 재편.
- **장비/인력공급사**: 선정되면 배차초안이 준비돼 1클릭 발송 + 단가 프리필. 다음은 배차+서류묶음 통합 발송.
- **협력사**: 부모가 대행등록+링크로 온보딩, 자식은 로그인해 본인분만 관리·정산 열람. 다음은 자가로그인 승인 경로 완결.
- **작업자**: 코드 로그인·출근·작업확인 — 유지. 워치 낙상감지 실기기 검증 후 Enforce.
