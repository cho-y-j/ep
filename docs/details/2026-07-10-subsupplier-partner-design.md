# 장비업체의 협력사(하위공급사) 붙이기 — 요구사항 정리 (진행중)

> 2026-07-10 대화로 모은 요구사항. **아직 설계 확정/구현 전.** BP노출 등 미정 항목 있음.

## 문제

장비업체(인력업체 포함)는 BP사와 계약해 장비·인력을 공급한다. 그런데 실제로는 **본인 장비·인력 외에 협력사의 장비·인력도 등록해서 함께 BP에 넣는다.**
BP는 계약된 장비업체에 정산 → 장비업체가 수수료를 공제하고 협력사에 재정산하는 구조.
→ "장비업체의 협력사를 어떻게 붙이는 게 가장 현명한가?"

## 결정된 요구사항

### 1. 입력 3경로 전부 필요 (택1 아님)
1. **협력사 로그인**: 협력사가 직접 로그인해 본인 장비·인력·서류를 입력/관리 (장비업체 관리자 페이지와 유사).
2. **장비업체 대행 등록**: 장비업체가 협력사 몫까지 대신 등록.
3. **링크 업로드**: 장비업체가 알림톡/메일로 링크 발송 → 협력사가 순서대로 서류 업로드. 잘못 올리면 보완요청 루프.

### 2. 소유 모델 (핵심 원칙)
- **협력사 = 하위공급사 회사(V77) 하나. 자원 소유는 항상 그 협력사.**
- 소유(누구 것)와 입력경로(누가 넣었나)를 **분리**. 위 3경로는 같은 소유 모델에 쓰는 얇은 입구.

### 3. 심사/빠꾸 (계층화)
- 장비업체가 **협력사 서류를 심사·반려(빠꾸)** 할 수 있어야 함.
- 현재 보완요청/심사는 **BP→공급사로 하드코딩**(BP/ADMIN만 생성 가능). → 이걸 **계층 인지(심사자↔피심사자 = 부모↔자식)**로 일반화해 재사용. "장비업체 = 협력사의 미니-BP".

### 4. 정산 (대폭 단순화)
- **시스템은 수수료를 계산·표기하지 않는다.** 실제 수수료 공제는 오프라인에서 각자 알아서.
- 필요한 것: **소유자별 "일한 것 정산 요약"** — 월대/일대/OT 등 단가·금액. 각자 본인 자원 것만 봄.
- 스코프: 장비업체=본인+협력사 전체, 협력사=본인 것만 (V77 `selfAndChildren`).
- → 빌링 엔진이 아니라 **집계 리포팅 뷰** 수준.

### 5. BP 노출 — **미정**
- (가) BP는 장비업체 명의만 본다 (협력사 내부, 현재 V77 기본)
- (나) BP도 실소유 협력사를 본다 (투명)

## 재사용 가능한 기존 자산 (코드 확인됨)
- **V77 하위공급사**: `companies.parent_company_id`(1단계 자기참조, 장비공급사만 부모), 자식에 독립 로그인 부여, 배차에 `supplier_company_id`(대외=부모) + `sub_supplier_company_id`(실소유 자식) 기록.
- **document_collection (V75, `com.skep.collection`)**: 무로그인 토큰 링크로 `sort_order` 순서대로 서류 업로드 → PDF 병합/이메일. `PublicCollectionController`.
- **document_supplement 보완요청 (`com.skep.supplement`)**: OPEN→RESOLVED(업로드 시 자동 close). 단 생성권한 BP/ADMIN 고정.
- **배차 단가**: `DispatchedEquipment.daily_price/ot_daily_price/monthly_price/ot_monthly_price` + `sub_supplier_company_id`(소유자) 가 같은 row. `QuotationRequest.work_period_*`, V67 field_deployment_rates, V47 OT단가.
- **정산 도메인/수수료 컬럼**: 없음 (신규).

## 2026-07-10 요구사항 재정리 (사용자 4개 항목 + 실행 방식)

**실행 방식: 1·2번 = 바로 기획→구현 / 3·4번 = 조사 후 대화.**

1. **장비공급사(인력공급사) ↔ 협력사 관계 명확히.** (위 "소유 모델" 뼈대 = V77 하위공급사)
2. **정산 뷰**: 장비공급사가 BP에 투입 시 일대/월대 등 정해진 투입조건에 따른 정산. **협력사도 본인 것이면 로그인해서 볼 수 있다.** (수수료는 시스템 미표기 — 오프라인)
3. **[조사 후 대화] 작업자/장비 운전수 모바일 앱 + 스마트워치 전면 점검.**
   - 스마트워치: 초기 착용 시 본인 맥박 등 **베이스라인 학습** → 실제 작업 중 **주변인 + 서버로 문제(이상징후) 알림**.
   - **특허 중 핵심**: 실제 문제 발생 시 **서버로 알려, 긴급상황에서 본인 주변인에게 알려 빠르게 조치**받는 것.
   - 요청: 모든 기능 점검 + 수정/개선/좋은 아이디어 제안.
4. **[조사 후 대화] 전체 UX 개선.** 사용자들이 더 쉽게 접근/사용할 수 있는지 전반적 개선안.

### 1·2번 구현을 위한 기본안 (미결정 → 기본값으로 진행, 사용자 veto 가능)
- BP 노출: **장비업체 명의만**(협력사 내부) — 노출 옵션은 후순위.
- 정산 기준: **계약 투입조건(일대/월대·OT 단가)** 우선. 실제 출역량 반영은 2차.
