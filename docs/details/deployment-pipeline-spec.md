# 자원 투입 파이프라인 — 확정 스펙 (canonical)

> 사용자 확정(2026-07-11). 장비/인력 1건(자원)의 **생애 = 서류 → 검사 → 투입대기 → 투입 → 작업 → 거래내역서**.
> 이 문서가 투입 흐름의 **단일 기준**이다. 재확인 없이 이 스펙을 따른다.

## 파이프라인 (단계 · 주체 · 상태)

| # | 단계 | 주체 | 동작 | 상태 |
|---|------|------|------|------|
| 1 | **서류 등록** | 공급사/협력사 | 장비·인력 서류 업로드 | 등록됨 |
| 2 | **검증** | 공급사/협력사·시스템 | 만료일 + 진위여부(OCR/정부API) 확인 → **서류 확인 / 반려(빠꾸)** | 확인 / 반려 |
| 3 | **BP 전달** | 공급사→BP | 확인 완료 서류를 BP사에 전달 — 목적: 투입 전 **장비검사** + 인력 **안전검사** + 배치 전 **건강검진** 신청 | BP 전달됨 |
| 4 | **BP 처리** | BP | 서류 확인 + **작업계획서** 작성 + 장비검사·안전교육 **신청** + **검사 날짜 통보** | 검사 신청/일정 통보 |
| 5 | **검사 승인 → 투입 대기** | BP·검사기관 | 검사 승인되면 자원이 **"투입 대기"** 상태 | **투입 대기** |
| 6 | **투입 요청** | BP(→공급사) | 실제 작업 투입 요청 (**구두 또는 웹**) | 투입 요청됨 |
| 7 | **실제 투입 + 정산율 확정** | 장비 임대사업자(공급사) | 실제 투입하며 **일대/월대 선택 + 단가 입력** → 실제 투입. **정산율은 이 시점에 확정** | 투입됨 |
| 8 | **작업 수행** | 작업자/운전자 | 투입 시점부터 **출근 체크 + 연장근무(OT) 시간** 기록 → **작업확인서** | 작업 중 → 작업확인 |
| 9 | **거래내역서** | 공급사/협력사 | 작업확인서 기반 **거래내역서(월별 또는 날짜지정)** 출력 | 정산/청구 |

## 확정 규칙 (재확인 금지)

- **정산율 확정 시점 = 실제 투입(7단계)**. 견적이 아니라 투입할 때 일대/월대 + 단가가 정해진다.
- **정산 계산**: 일대 = 일대×근무일수 + OT일수×OT일단가 / 월대 = (월대 ÷ 25)×근무일수 + OT일수×OT월단가. (÷25 = 월 근무일수). 근무일수·OT는 작업확인서에서 자동 파생(수동 우선). → `SettlementCalculator`.
- **거래내역서(9단계) = 작업확인서 + 정산의 결합 문서**, 월별 또는 임의 날짜범위로 출력. (현 정산 요약과 별개의 "명세/청구" 산출물 필요 여부는 코드 매핑으로 확정.)

## 조합(차량+조종원 매칭) — 확정 (2026-07-23, R1 구현)

- **원장 = `equipment_default_operators`** (장비 1 : 조종원 N, priority = 교대 순번. 행이 없으면 `equipment.operator_person_id` 단일 폴백). 이 조합으로 서류수집·검사·투입·정산이 이어진다.
- **관리 주체 = 소유 공급사(자기+직속 자식) + ADMIN** — BP 매칭 허용 제거(조회는 유지). 변경은 audit_logs(`EQUIPMENT_DEFAULT_OPERATORS_CHANGED`, 전/후 person_ids)에 기록.
- **R1 판정 = 파생(저장 없음)**: `GET /api/resources/equipment/{id}/deploy-check-combo?siteId=` — 기존 4게이트(DeployCheckService)를 장비 1회 + 조종원 N회 합성. `combo_ready = 장비 ready AND 조종원 전원 ready`(0명이면 장비 단독). 접근권한은 **장비 접근권만** 검사(조종원은 내부 판정 — 장비가 보이면 조합 판정도 보임).
- **단계 스냅샷(R2~R4 예정)**: 검사/투입/정산 단계 행에 `combo_equipment_id` 스냅샷을 남겨 조합 단위 추적. R1 은 판정+표시만(마이그레이션 없음).
- **재검사 이원화 해소(R1 동반)**: 점검 재발행(`ResourceCheckService.issue`) 시 같은 자원·같은 check_type 의 기존 REJECTED 를 자동 CANCELLED — readiness/파이프라인/작업계획서 제출 술어("전부 APPROVED, CANCELLED 통과")와 deploy-check 판정이 재검사 승인 후 일치.

## 협력사(장비 협력사, V77 하위공급사) — 확정

- **대행 = 전체 CRUD**: 부모(장비공급사)가 자식(협력사)의 **사업자정보·인력·장비를 등록 + 수정 + 삭제 모두** 대행 가능. (현행: 생성만 가능, 수정 403 → 수정권 확장 필요.)
- **자가 로그인 + 부모 승인**: 협력사가 **스스로 가입/로그인**하고, **부모(장비공급사 관리자)가 승인**해 활성화한다. (현행: self-signup enabled=false → 시스템 ADMIN 게이트 → 부모 승인 경로 신설 필요.)
- 협력사는 로그인 후 본인 자원을 관리하고, **본인 투입분 정산을 열람**한다(소유자별·본인 격리 — 이미 동작).
- 온보딩 3경로: (a) 협력사 자가 로그인 업로드, (b) 부모 대행 등록, (c) 링크 발송→협력사 업로드. (b)(c) 구현됨, (a)는 자가로그인+부모승인 완결 필요.

## 구현 작업 영역 (이 스펙에서 파생)

- **A. 협력사 대행 전체 수정권** — `ensureCanModify`를 selfAndChildren로 확장(Equipment/Person/Company).
- **B. 협력사 자가로그인 + 부모 승인** — 부모가 자기 자식회사 유저를 승인/활성화하는 경로 신설.
- **C. 검사 파이프라인(3~5단계)** — BP 전달 → 장비검사·안전검사·건강검진 신청 → 검사 날짜 통보 → 검사 승인 → **"투입 대기" 상태**. (코드 매핑으로 exists/gap 확정 후 구현.)
- **D. 거래내역서(9단계)** — 작업확인서 + 정산 결합, 월별/날짜지정 출력. (코드 매핑으로 기존 명세 유무 확정 후 구현.)
