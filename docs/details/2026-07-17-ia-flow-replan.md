## 0-A. UX 대원칙 (2026-07-17 사용자 — 전 화면 적용, 어드민 포함)

1. **무조건 사용이 편해야 한다** — 클릭 최소, 역할별 첫 화면 = "오늘 내가 할 일"이 바로 보임.
2. **보기 좋은 디자인 + 차트 대시보드** — 표 나열이 아니라 카드·차트로 **한눈에 상황 파악**.
3. **작업자도 1급 사용자** — 운전원·신호수 등 **역할에 맞는 앱 동작** + **내가 일한 내역(작업확인서 기반 일일/월간, 서명 상태, OT 시간)을 앱에서 본인 확인** 가능.
4. **위치·안전·혼잡도 대시보드** — 작업자 위치(구역 단위)·안전 상태·**시간대별 혼잡도**를 관제 차트로 → 점심시간 조정 등 현장 계획의 데이터 자산화. (정밀 실내측위는 비콘 없인 한계 — 구역(공구·레벨) 단위 분포는 출근·지오펜스·작업위치 기록으로 가능, 정직 표기)

## 0. 한 줄 목표

BP가 **웹에서 서류를 바로 열어 승인하고, 그 자리에서 작업계획서와 검사·교육·검진 날짜 통보까지** 끝내는 원스톱 흐름을 만들고, 메뉴를 역할당 5허브로 재편하며, 견적·서류·투입·정산이 **각각 단독으로도** 돌아가게 한다.

### 3.2 디커플링 (견적/서류/투입/정산 각자 단독) — 2026-07-17 사용자 확정

- **확정 원칙**: 견적→심사→투입 전체 연결은 이상적 경로로 **그대로 유지**하되, 현장 현실(견적은 견적대로, 실제는 서류심사부터)에 맞춰 **서류심사부터 시작하는 경로를 1급 동선**으로 노출. **견적서는 업체 선정의 근거자료로 보존**(선정 이력에서 참조 가능해야).
- 이미 단독 가능: 서류심사·작업계획서·투입요청·투입대기 → IA에서 "견적 없이 시작" 동선 노출.
- 신규: **투입 확정(FieldDeployment.accept) 시 정산 원천 행 생성** → 견적을 안 거친 투입도 투입보드·정산에 잡힘. (`FieldDeploymentService.accept:131` 훅)

### 3.3.1 ★계약(Contract) 단계 신설 — 단가의 진짜 원천 (2026-07-17 사용자 교정)

**현장 실제 흐름**: 전화 견적요청 → 견적서 메일(참고) → **현장 소장과 계약서 작성 = 단가 확정 시점** → 투입 날짜 잡히면 서류 발송 → BP 작업계획서 → 투입 지시(날짜) → 투입 → 정산(계약 단가).

- 시스템에 "계약" 단계가 없어 단가 확정의 앵커가 부재했음 → **계약 레코드 신설**: BP↔공급사·현장·장비종류/규격·**5분류 단가 세트**·기간·계약서 파일(스캔 첨부).
- 생성 경로 2가지(디커플링 원칙): ⓐ 시스템 견적에서 선정 시 자동 생성(견적 링크=근거자료) ⓑ **전화/메일 견적이었어도 계약만 직접 등록**(견적 없이 — 현장 대부분의 케이스).
- 정산·투입 단가는 **계약에서** 온다(견적 스냅샷 아님). 투입 지시(날짜 통보) 단계도 흐름에 명시.

### 3.6 ★작업계획서 자동화 (핵심 가치 — 2026-07-17 사용자 통찰로 승격)

**문제의 본질**: 차량·인력이 바뀔 때마다 BP가 작업계획서를 새로 써야 해서 **기존 업체를 못 바꾼다**(전환비용=락인). 이 프로그램의 결정적 가치 = 이 전환비용 제거. 서류검증·OCR로 자원 데이터가 이미 시스템에 있으므로 실현 가능성 높음.

- **L1 자동 생성**: 등록 자원 데이터(OCR 추출·검증된 차량번호·규격·면허·보험·검사일)+현장 정보로 계획서를 자동 채움 — 수기 재입력 0. (P0의 "승인→계획서 프리필"이 진입점)
- **L2 교체 델타 — 현장 단위 규칙 (2026-07-17 사용자 확정)**:
  - **같은 현장 + 기검증 자원**(반입검사·인력관리 완료): **업체변경 신청서 한 장으로 재투입 가능** → L2a = 현장 내 기검증 자원 풀에서 교체 후보 선택 → **업체변경 신청서 자동 생성(DOCX)** → 서명 → 끝. BP가 기존 현장 자원을 선호하는 현실을 그대로 시스템화 — 신규 업체도 한 번 반입되면 동등해져 락인 해제.
  - **다른 현장이면 신규**(같은 SK라도 관리 현장이 다르면): 서류 심사→반입검사→안전교육 전부 다시 = L1 자동생성 + 검사·교육 일정 자동 요청으로 부담 최소화.
  - 교체 시점 셋 다 실재(당일 긴급 고장·결근 / 사전 계획 / 신규 업체 추가) → 긴급용 모바일 동선 + 사전검증(L3) 모두 필요.
- **주기 관리(신규 발견)**: 반입검사 **월 1회**·건강검진 **6개월** 재실시 — 주기가 현장마다 달라 **BP가 현장 프로필에서 설정**. 만료관리와 동일 패턴으로 재검·재검진 임박 알림.
- **L3 교체 사전검증**: 교체 후보 선택 시 서류·만료·점검(주기 포함) 게이트 자동 체크(readiness 재사용) → "바로 재투입 가능 / 반입검사 기한 초과 / 서류 2건 부족" 즉시 표시.

- 구두 승인 건은 기록 로그(누가·언제 기록)가 자동 보존(분쟁 대비). BP가 추후 프로그램에 합류하면 구두 승인 건을 소급 승인으로 전환 가능.

**설계**: ① 공급사(장비업체)가 자원 단위로 **[기통과 완료 체크]**("이 현장에 심사·반입검사·교육·검진 이미 완료" 자가 신고, 현재 작업 중 표시) → ② **BP가 목록으로 받아 일괄 승인**(자원별 체크박스, 한 화면에서 끝 — 신규 파이프라인 재통과 없음) → ③ 승인 즉시 투입 상태 → 오늘부터 일일 확인서·정산 가동.
- 완료일자(반입검사일·교육일·검진일)는 **선택 입력** — 적으면 주기 관리(월1회·6개월 재검 알림)가 그날부터 이어짐, 비우면 승인일 기준.
- 소급 건은 **"소급 승인" 라벨**로 정상 파이프라인 통과 건과 구분 표시(감사 추적).
- L2a의 "현장 내 기검증 자원 풀"에 소급 건 포함 → 교체 후보로도 즉시 사용.
- **첫날 사용 시나리오**: 자원 등록(서류는 있는 만큼) + 기통과 체크 + 계약 단가 등록 → 투입 → 일일 확인서 기록 + BP 서명 → 월간·정산. 견적·심사 디지털화는 그 다음 건부터 자연 유입.

## 4-B. ★IA 시정 확정판 (2026-07-18 사용자: "그룹핑≠간소화, 정신없다" — P2a 반성 후 재설계, 전권 위임)

**파이프라인 드릴다운·필터 (사용자 확정)**: 자원 파이프라인에서 단계/항목 **클릭 시 해당 건만 정렬**되어 한눈 파악 + 필터바(장비별·업체(공급사/협력사)별·현장별) + 장비 1대 선택 시 그 장비의 파이프라인만(타임라인 뷰). 다른 목록 화면에도 동일 패턴 점진 적용.

**웨이브**: P4a 안전 상황판 → P4b 메뉴 다이어트+유틸 소메뉴 → P4c 파이프라인 드릴다운·필터 → **P4d (2026-07-18 사용자 추가 지적)**:

## 4-C. ★P5 워치 스마트 안전 — 감지→확산→발견→기록 대응 체인 (2026-07-18 사용자: "프로그램의 핵심")

**기존 자산(워치 검토 실사)**: 10분 개인 학습(CalibrationActivity)·5단계 적응 센서·낙상 판정(FallDetector 테스트 3/3)·45초 자가취소(AckAlert)·동료 경보 수신(P2pAlertActivity)·BLE(BleAlertService)·바이탈 5분 전송(/api/field-auth/sensor)·관제 WS.

**★뇌심혈관 우선 (2026-07-18 사용자: 현장 최다 사고=뇌출혈·혈압) — 3겹 설계**:
- **1겹 작업 전 혈압 체크인**(최우선·진짜 예방): 현장 커프 혈압계 → 출근 게이트로 측정 기록(블루투스 연동 or 수기) → 임계 초과 시 고소·고강도 배제 권고+관리자 통보+기록(증거사슬).
- **워치 혈압 보조 트랙(2026-07-18 조사·사용자 질문)**: 갤럭시 워치5 혈압 측정 **가능**(삼성 헬스 모니터·식약처 승인·오차 5mmHg 이하) — 단 ①커프 보정 최초 3회+**28일마다 재보정**(현장 커프 어차피 필요) ②**보정 사람별**(공용 돌려쓰기 불가→개인 지급 시만 유효) ③수동 스팟 측정(상시 감시 불가·백그라운드 X). 2025.2~ Samsung Health Data SDK 혈압 타입 지원+Health Connect 양방향 동기화 → **폰앱이 동의 기반 Health Connect로 혈압 읽는 표준 경로 열림**(한국 SHM→Health Connect 실동기화 여부는 실기기 검증 필요·체크리스트 추가). 활용: 고위험군 낮 스팟 체크 알림("점심 후 혈압 재세요")→자동 수집 시도·폴백 수기. **커프 체크인 게이트가 여전히 주 경로**(보정 의존성+상시 불가+증거사슬).
- **2겹 고위험군 태깅**: 건강검진 서류(기보유) 기반 고혈압·심혈관 위험군 → 워치 우선 착용·심박 임계 하향·휴식 빈도↑·폭염 제한 강화·과로 경고(연속 야간·장시간 — 근무 데이터 결합).
- **3겹 골든타임**: 뇌출혈=발견 속도가 예후 — 워치는 발생 순간(급성 심박 급변·활동 제로+무반응)을 잡아 P5b 체인 즉발. 혼자 쓰러지는 시나리오에서 분 단위 단축이 워치의 존재 이유.

**P5a 감지 고도화**: 개인 베이스라인(안정/작업 심박 대역) + 맥락 판정(심박↑+활동량↓=쓰러짐 전조 / 폭염단계+체온↑+심박↑=열사병 전조→선제 휴식 지시) + 서버 2차 확정 + 오탐 시 개인 임계 자동 보정(학습 루프) + 관제 워커별 바이탈 스파크라인. **포지셔닝: 의료 판단 아닌 조기 경고 보조(법적)**.
**P5b 대응 체인**: 미취소 긴급 → 근접 동료 3명(BLE RSSI > GPS > 같은 작업조) → [제가 갑니다] 응답 추적(관제 표시) → 60초 무응답 시 반경 확대+관리자+알림톡. **파인드미 비콘**: 피재자 폰 사이렌 최대+플래시 스트로브+워치 진동 지속, 응답자 폰에 거리·방향+BLE 근접 게이지.
**[결정 2026-07-19] 데드맨≠동료 체인**: 무수신(watch_offline·CAUTION)은 배터리/미착용 오경보 가능성 때문에 동료 3인 긴급 호출 미발동 — 본인 확인→5분 관리자 에스컬만. 동료 체인은 EMERGENCY(낙상·SOS·릴레이 수신) 전용. 경보 피로 방지=긴급 신뢰 유지.

**P5c 무신호 릴레이+기록**: 워치 BLE SOS를 임의 SKEP 폰이 수신→대리 중계(터널·지하) + **골든타임 타임라인**(감지→통보→응답→도착→조치 타임스탬프)→이행 보고서 "긴급 대응 이력" 섹션.
**★P5-전력 설계 원칙 (2026-07-18 사용자: 10시간 작업 배터리가 최우선) — "침묵이 정상, 예외만 보고"**:
- **실시간 스트리밍 폐기 → 예외 기반**: 판단(베이스라인 비교·낙상)은 워치 온디바이스, 평상시엔 거의 무전송. 3색 상태머신: 🟢평상=심박 1분 듀티·**10~15분 BLE 묶음 전송**(20~30B 압축)·GPS 꺼짐(~3~4%/h) / 🟡주의(고위험군·폭염·이상경향)=5분 묶음(~5%/h) / 🔴경보=실시간 스트림+GPS+비콘(사건은 분 단위라 무제한). **목표 10h ≈ 40% 소모**(워치5 410mAh·실기기 캘리브레이션 필요).
